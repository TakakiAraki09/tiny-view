# アーキテクチャ書: エージェント駆動開発に耐えるリポジトリ

このアーキテクチャの設計原理を説明する。SKILL.md が「何をするか」なら、この文書は「なぜそうなっているか」であり、立ち上げ時の個別判断（ゲートを hard にするか soft にするか、何を文書化し何をしないか）に迷ったときの判断基準を与える。

## 1. 中心思想: 品質保証を人間レビューから構造へ移す

人間が書く伝統的なリポジトリは、品質保証の最終ラインを人間レビューに置く。書き手がプロジェクトの文脈を記憶しているため、文書が薄くてもレビューは機能する。

エージェント駆動開発ではこの前提が崩れる。エージェントは (a) 会話ごとに文脈を失い、(b) 並列に複数動き、(c) 人間より遥かに速く変更を量産する。人間レビューを最終ラインのままにすると、レビューが律速になるか、レビューが形骸化してリポジトリが劣化するかの二択になる。

解決は、**人間が一度だけ高品質な判断（原則・KPI・スコープ）を行い、それを文書と機械ゲートに固定して、以後の全エージェント作業に自動適用させる**ことだ。各層の役割:

| 層 | 守るもの | 検査主体 |
|----|---------|---------|
| 絶対条件 / KPI | プロダクトの方向性 | 人間（一度だけ策定） |
| PRD | 仕様の正 | エージェント（参照） |
| CLAUDE.md | 毎作業の判断基準 | エージェント（毎回読む） |
| *.test.md + テスト | 仕様と実装の一致 | CI |
| CI ゲート | KPI・品質の劣化 | CI |
| リリース2段階 | 不可逆操作 | 人間（トリガーのみ） |
| worktree hook | main の清浄さ | hook（機械） |

人間の役割は「策定」と「不可逆操作の承認」に縮約され、それ以外は構造が守る。これがこのアーキテクチャの全てで、以下の各層はこの思想の具体化にすぎない。

## 2. 第一層: 絶対条件と KPI

### 絶対条件は「検証可能な禁止/保証」でなければならない

絶対条件の機能は、将来のエージェント（と人間）の提案を**機械的に却下できる**ことにある。「シンプルであること」「高速であること」は価値観であって絶対条件ではない — 却下の根拠として運用できない。良い絶対条件は次の形をとる:

- 禁止形: 「バックグラウンドプロセスを持たない」「ネットワークポートを listen しない」「ビルド時に外部サービスへアクセスしない」
- 保証形: 「終了時に全状態を破棄する」「単一バイナリで配布する」「入力はすべて X 形式に正規化してから処理する」

判定テスト: **その条件への違反を、コードレビューまたは CI で機械的に指摘できるか？** できないなら言い換えるか、絶対条件から外して KPI かビジョン文に降格する。

個数は 3〜7 個。多すぎると毎回のチェックが形骸化し、少なすぎると防衛線にならない。

### KPI は優先順位を固定し、hard/soft を分類する

KPI で重要なのは数値そのものより**優先順位の固定**だ。「起動速度 > メモリ > 機能量」と順位が固定されていれば、エージェントはトレードオフ判断（この機能はコアに入れるべきか）を自律的に下せる。順位がないと、エージェントは依頼ごとに目先の要求を優先し、KPI が静かに劣化する。

各 KPI は次のどちらかに分類する:

- **hard**: CI 環境で安定して計測できる → fail 閾値を設けてゲート化。ただし目標値そのものを閾値にせず、ランナーのノイズを見込んだマージンを乗せる（目標 150ms なら fail は 200ms など）。flaky なゲートは全ゲートの信頼を毀損するため、厳しすぎる閾値は逆効果
- **soft**: 環境依存で CI では正確に測れない（GUI 必須のメモリ計測など）→ CI では参考値レポートに留め、**ローカルでの正式な計測手順を文書化**する。「測れないから何もしない」と「測れないことを文書化して手動手順を定める」の差は大きい

### 「コアに入れない」原則

KPI を一度決めたら、それを悪化させる機能はコアに入れず、テンプレート・プラグイン・オプションに外出しする。この原則を CLAUDE.md に明文化しておくと、機能追加依頼に対してエージェントが「コアではなく外出しで実装します」という正しい第一案を出せるようになる。

## 3. 第二層: PRD と CLAUDE.md の二層文書

### なぜ二層必要か

一層（CLAUDE.md だけ）だと、仕様の全量を書けば毎回のコンテキストコストが膨らみ、要約だけ書けば仕様の正が存在しなくなる。二層に分け、役割を非対称にする:

- **PRD**: 仕様の全量。人間が読み、エージェントが「解釈に迷ったとき」参照する。更新頻度は低い
- **CLAUDE.md**: 毎作業で読まれる前提の圧縮版。チェックリストと表が中心。**PRD にない新情報を含めてはならない**（含めた瞬間、正が二箇所になりドリフトが始まる）

衝突時は PRD が正、と両文書に明記する。

### §番号は文書の API である

PRD の全セクションに §番号を振るのは見栄えのためではない。§番号は、コードコメント・テスト仕様・CI 設定・issue・コミットメッセージから仕様を指す**安定した番地**になる。「PRD §9.10 に従い debounce は 100ms」とコードに書ければ、その行の変更可否の判断は PRD を引くだけで済む。番地のない文書への参照（「PRD の watch のところ参照」）は文書改編で壊れる。

セクションの増減で番号を振り直さないこと。欠番は許容し、挿入は枝番（§9.10.1）で行う。

## 4. 第三層: テスト⇔仕様の双方向トレーサビリティ

実装ファイルごとに `<モジュール>.test.md` を隣置きし、各テストを1行ずつ次の表に載せる:

```markdown
| 検証事項 | 内容 | 備考 |
|---------|------|------|
| マーカー置換は1回のみ | テンプレート内の2個目のマーカーが置換されないこと | PRD §13 / `render_replaces_only_first` |
| 閉じタグ注入の無害化 | 入力中の `</script>` がエスケープされること | issue #29 / `build_literal_escapes_script_close_tag` |
```

これが与える性質:

- **順引き**: 仕様変更時、影響テストを表から特定できる
- **逆引き**: テストが落ちた/消されたとき、どの仕様への違反かが分かる。エージェントが「邪魔なテストを消して通す」という典型的な不正をしたとき、diff に現れた `.test.md` の行削除が「仕様 §X を放棄した」と読めるため、レビューで機械的に検出できる
- **意図の保存**: 「なぜこのテストがあるか」（issue 番号・障害の記憶）がコードより長持ちする

コストは1テストあたり1行の表更新であり、テストを書いたエージェント自身に同じターンで書かせれば維持できる。後からまとめて書かせると形骸化する — CLAUDE.md に「テスト追加時は同一コミットで .test.md を更新する」と入れること。

## 5. 第四層: CI ゲート

### 警告ゼロポリシーの理由

「lint 警告は新規分だけ増やさない」という運用は人間にも難しいが、エージェントには不可能に近い — エージェントは既存警告と自分の警告を区別する文脈を持たない。基準を「ゼロ」に固定すると判断が不要になる: 警告が1件でもあれば自分の責任で直す。これは format チェックも同じで、機械的に判定できる基準だけがエージェント相手に機能する。

### ゲート構成の標準形

| job | 目的 | hard/soft |
|-----|------|-----------|
| build + test（全対象プラットフォーム matrix） | 仕様との一致 | hard |
| lint（警告=エラー） + format | コード品質の床 | hard |
| 最低サポートツールチェーン検証 | 依存更新による静かな要求引き上げの検知 | hard |
| KPI 計測（性能・サイズ等） | KPI の劣化検知 | KPI 分類に従う |
| 統合/E2E | 実環境での動作 | 環境が安定しないなら soft + ローカル正式手順 |

加えて、同一ブランチの古い CI 実行を自動キャンセルする設定を入れる。エージェントは push 頻度が高く、これがないと CI キューが詰まって全員のサイクルが遅くなる。

### soft ゲートの規律

soft（continue-on-error / 警告のみ）にした項目には、必ず**ローカルでの正式検証手順**を README か PRD に書く。soft ゲートは「CI が緑でも保証されていない」項目なので、何が保証外かをリポジトリ内で明示しないと、緑＝安全という誤解が蓄積する。

## 6. 第五層: リリース自動化と不可逆操作のゲート

リリースフローの設計原理は「**可逆な操作は自動化し、不可逆な操作だけを人間ゲートにする**」。

1. main への push → リリースツールが**リリース PR**（バージョン bump + CHANGELOG + lockfile 同期）を自動で開く・更新する。マージしても公開はされない（可逆）
2. リリース PR をマージ → bump が main に乗るだけ（まだ可逆）
3. 人間が CI の手動トリガーで publish を実行 → レジストリ公開 + タグ + リリース作成（不可逆はここだけ）

この3段のうちエージェントが関与してよいのは「適切な Conventional Commit を積む」ことだけだ。バージョン・タグ・lockfile はリリースツールの所有物とし、手動 bump を禁止する。理由は二つ: (a) 手作業とツールの競合は lockfile ドリフト等で publish を壊す、(b) 「コミット規約さえ守ればリリースは正しく動く」という単純な契約のほうが、エージェントに守らせやすい。

bump 規則（fix→patch, feat→minor, 破壊→major, それ以外のデフォルト）は採用ツールの実際の挙動を確認して CLAUDE.md に表で書く。ツールのデフォルトは製品によって微妙に違い、「chore はリリースされない」といった思い込みが事故になる。

## 7. 隔離層: worktree 強制とオーケストレーション

### worktree 強制 hook

エージェントの編集ツール実行をフックし、primary worktree（main のチェックアウト）上ならブロックして、作業用 worktree の作成手順をエラーメッセージで案内する。

これは「エージェントに worktree を使うよう指示する」のとは違う。指示は文脈圧縮や長い会話で忘れられるが、hook は忘れない。**守らせたい規律は、指示ではなく機械的強制に変換できないか常に考える** — このアーキテクチャ全体を貫く原理の最小例がこの hook である。

副次効果として、worktree 単位の隔離は並列エージェント実行（複数 issue の同時処理）の前提条件にもなる。

### オーケストレーションスキル

リポジトリに2つのワークフロー定義を置く:

- **単一タスク→PR**: worktree 作成 → 実装 → ローカル検証（test/lint/format）→ 規約準拠 commit → ドラフト PR → 独立レビュー → 合格なら Ready 化、不合格なら修正ループ（上限付き、行き詰まり時は人間にエスカレーション）
- **複数 issue 一括処理**: ラベルで issue を収集 → 件数と内容を人間に提示して Go を取る（確認ゲート）→ 1 issue = 1 エージェント = 1 worktree で並列実行 → 結果を表に集約（Ready / Draft / Failed）。1件の失敗で全体を止めず、部分成功を許容する

どちらも「PR は必ずドラフトで作り、検品を通ってから Ready にする」「修正ループに上限を設ける」が骨子。無限ループ防止と、未検品 PR が人間のレビューキューに流れ込むことの防止である。

## 8. アンチパターン集

- **雛形だけ置く**: 空の PRD テンプレ、0行の .test.md 表、TODO だらけの CLAUDE.md。実例が1つもない文書は誰も埋めない。立ち上げ時点で各文書に最低1つの実内容を入れる
- **CLAUDE.md に仕様を直接書き足す**: 正が二箇所になりドリフトする。仕様は PRD に書き、CLAUDE.md には圧縮を書く
- **目標値をそのまま CI の fail 閾値にする**: ランナーのノイズで flaky になり、エージェントが「このゲートは無視してよい」と学習する。マージンを乗せる
- **絶対条件が価値観**: 「シンプルに保つ」は却下根拠として運用できない。検証可能形に言い換える
- **エージェントにバージョンを触らせる**: lockfile ドリフトや二重タグで publish が壊れる。リリースツールの所有物にする
- **soft ゲートの放置**: 「CI 緑＝安全」の誤解が蓄積する。soft 項目には必ずローカル正式手順を文書化する
- **規律を指示で守らせる**: 指示は忘れられる。hook・CI・ツール所有権など機械的強制に変換する
