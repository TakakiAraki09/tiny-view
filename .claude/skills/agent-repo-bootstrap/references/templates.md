# 生成物テンプレート集

各テンプレートの `<...>` はヒアリング結果で置換する。**構造は維持し、内容はプロダクト固有に書き切る**こと。プレースホルダや TODO を残したまま完了としない（唯一の例外: ユーザー未確認の草案に付ける `<!-- 要確認: ユーザー未承認 -->`）。

## 目次

1. docs/PRD.md 骨子
2. CLAUDE.md 骨子
3. *.test.md 骨子
4. CI ゲート構成チェックリスト
5. オーケストレーションスキル骨子（単一タスク→PR / 複数 issue 一括）

---

## 1. docs/PRD.md 骨子

```markdown
# <プロダクト名> PRD

このドキュメントが仕様の single source of truth。CLAUDE.md と矛盾したら本書が正。

## §1 概要
<一行定義。「何でないか」も一文で>

## §2 解決する課題
<既存手段の何が問題で、何を置き換えるのか>

## §3 絶対条件
<検証可能な禁止/保証の形で 3〜7 個。各条件に違反例を1つ添える>

| § | 条件 | 違反の例 |
|---|------|---------|
| §3.1 | <禁止/保証> | <これをやったら違反> |

## §4 KPI
優先順位は固定。下位 KPI のために上位 KPI を犠牲にしない。

| 優先 | KPI | 目標値 | 分類 | 検証方法 |
|------|-----|--------|------|---------|
| 1 | <例: 起動時間> | <数値> | hard | CI <job名> (fail閾値: <マージン込み>) |
| 2 | <例: メモリ> | <数値> | soft | <ローカル正式手順への参照> |

## §5 ユーザー像とユースケース
<主要ユースケースを入力→操作→期待結果の形で>

## §6 機能仕様
<機能ごとに §6.x を振る。入力・出力・エラー時挙動を明記>

## §7 MVP スコープ
**含む:** <列挙>
**含まない:** <列挙 — 将来のスコープ膨張への防衛線。理由も一言>

## §8 セキュリティデフォルト
<デフォルトで無効な権限と、明示的に有効化する手段>

## §9 リリースとバージョニング
<採用したリリース自動化の仕組み、bump 規則、2段階ゲートの手順>
```

## 2. CLAUDE.md 骨子

```markdown
# CLAUDE.md

このリポジトリで作業する際の指針。詳細仕様は `docs/PRD.md` を参照。矛盾したら PRD が正。

## プロダクト概要
<一行定義 + 最小の使用例>

## 絶対条件（破ってはならない原則）
実装提案・コードレビュー時は常にこれらに反していないかチェックする。
<PRD §3 の転記。各項目に PRD §番号を付ける>

## 最重要 KPI
優先順位は固定:
<PRD §4 の順位リストと目標値の表>

**<最上位KPI> を悪化させる機能はコアに入れない。** <外出し先: テンプレート/プラグイン/オプション等>に外出しする。

## コアに入れて良いもの / いけないもの
**Core OK:** <列挙>
**Core NG:** <列挙>

## 作業時のチェックリスト
新機能・依存追加・アーキテクチャ変更を提案する前に:
- [ ] KPI 目標を悪化させないか
- [ ] 絶対条件（§3）に反していないか
- [ ] コアではなく外出しできないか
- [ ] 永続状態を持ち込んでいないか
- [ ] テスト追加時、同一コミットで対応する *.test.md を更新したか

## リリースとコミット規約
コミットは Conventional Commits 必須。<採用ツール>がコミットメッセージから次バージョンを自動算出する。

| prefix | bump |
|--------|------|
| fix: | patch |
| feat: | minor |
| feat!: / BREAKING CHANGE: | major |
| その他 (chore:/docs:/ci:/...) | <採用ツールの実挙動を確認して記入> |

**バージョン / タグ / lockfile を手で触らない。** これらは<採用ツール>が所有する。

リリースフロー:
1. main に push → リリース PR が自動で開く/更新される
2. リリースしたくなったらリリース PR をマージ（この時点では公開されない）
3. <手動トリガー手順> で初めて publish + タグ + リリース作成
```

## 3. *.test.md 骨子

実装ファイル `src/<module>.<ext>` の隣に `src/<module>.test.md` を置く。

```markdown
# <module> テスト仕様

| 検証事項 | 内容 | 備考 |
|---------|------|------|
| <仕様を一言で> | <何が成り立つことを確認するか> | PRD §<番号> / `<テスト関数名>` |
```

規律: テストを追加・変更・削除するコミットは、同一コミットでこの表を更新する。備考列には必ず PRD §番号・issue 番号・テスト関数名のいずれか（できれば複数）を入れる。

## 4. CI ゲート構成チェックリスト

CI プラットフォームと言語に合わせて実装する。job 名は自由だが、役割の抜けがないこと:

- [ ] **build + test** — 対象プラットフォーム全部の matrix
- [ ] **lint** — 警告をエラー扱いにするフラグ付き。feature/オプションビルドがあるならそれも lint 対象に
- [ ] **format チェック** — 自動整形ツールの check モード
- [ ] **最低サポートツールチェーン** — 宣言した最低バージョンでビルドが通ることを毎 push 検証
- [ ] **KPI 計測 job** — PRD §4 の hard KPI ごとに: 計測 → 目標値レポート → マージン込み fail 閾値で判定。soft KPI は参考値レポートのみ + 正式手順への参照コメント
- [ ] **同一 ref の旧実行自動キャンセル** — concurrency 設定
- [ ] **リリース workflow** — リリース PR 自動生成（push トリガー）と publish(手動トリガーのみ)の分離。fork からの誤発火ガード（リポジトリ名チェック）

## 5. オーケストレーションスキル骨子

`.claude/skills/` に2つ置く。エージェントハーネスがスキル機構を持たない場合は `docs/WORKFLOWS.md` として置き、CLAUDE.md から参照する。

### 5a. 単一タスク→PR ワークフロー

```markdown
---
name: task-to-pr
description: 実装タスクを worktree 隔離 → 実装 → 検証 → commit → ドラフトPR → 検品 → Ready 化まで一気通貫で回す。実装からPR提出までを任される依頼で使う。
---

# 絶対原則
1. primary worktree で直接編集しない（hook でも強制されている）
2. PR は必ずドラフトで作成し、検品合格後に Ready 化する
3. 修正ループは上限 5 回。行き詰まったら人間にエスカレーション
4. CLAUDE.md の絶対条件・チェックリストを実装前に確認する
5. push / PR 作成など外部に出る操作の前に状態を確認する

# フロー
1. 前提確認: タスクの明確化、CLAUDE.md / PRD 読込
2. worktree 作成: `git worktree add ../<repo>-<task> -b <type>/<task>`
3. 実装（テスト追加時は同一コミットで *.test.md 更新）
4. ローカル検証: test / lint / format / build
5. Conventional Commit
6. ドラフト PR 作成
7. 検品: 絶対条件適合 / KPI 影響 / テスト妥当性 / 規約準拠 の4軸
8. 合格→Ready 化、不合格→3 に戻る（上限まで）
```

### 5b. 複数 issue 一括処理

```markdown
---
name: issue-batch
description: 特定ラベルの issue を収集し、1 issue = 1 エージェント = 1 worktree で並列に実装→PR 化する。複数 issue をまとめて処理する依頼で使う。
---

# 絶対原則
1. 実行前に issue 一覧を人間に提示し Go を取る（確認ゲート）
2. 1件の失敗で全体を止めない。部分成功を許容する
3. issue ごとに worktree / ブランチを隔離する
4. 完走は Ready、未完は Draft のまま残す
5. 結果は表（issue / 状態 / PR / 備考）に集約して報告する

# フロー
1. ラベルで issue 収集 → 一覧提示 → Go 確認
2. 同一ターンで全 issue を並列ディスパッチ（各エージェントは task-to-pr に従う）
3. 結果集約 → Ready / Draft / Failed の表で報告
```
