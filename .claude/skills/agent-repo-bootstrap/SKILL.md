# エージェント駆動開発リポジトリの立ち上げ

新規リポジトリを、**どのAIエージェントが何度作業しても品質が機械的に守られる**構造で立ち上げる。

## なぜこの構造か

エージェントは会話ごとに文脈を失う。人間のレビューだけを品質の砦にすると、エージェントの作業量が増えるほどレビューが破綻する。このアーキテクチャは品質保証を**文書と機械ゲートの連鎖**に移す:

```
絶対条件 + KPI（少数・検証可能）
  ↓ 展開
docs/PRD.md（§番号付き single source of truth）
  ↓ 圧縮
CLAUDE.md（エージェント向けチェックリスト）
  ↓ 参照
実装 + *.test.md（テスト⇔仕様の双方向トレース）
  ↓ 機械検証
CI（lint警告ゼロ + KPIゲート）
  ↓ 自動化
リリース（コミット規約→自動バージョン→2段階publishゲート）
  ⟂ 隔離
worktree強制hook + PRワークフロー
```

各層は上の層から**導出**される。だから最初のヒアリング（Phase 0）が最も重要で、ここを飛ばして雛形だけ置くと全層が形骸化する。設計原理の詳細は `references/architecture-book.md` を読むこと。

## Phase 0: ヒアリング（生成より先に必ず行う）

以降の全ファイルはここで決めた内容から導出される。揃えるものは5つ:

1. **プロダクトの一行定義と「何でないか」** — 位置付けを一文で。「何でないか」が後のスコープ防衛線になる
2. **絶対条件 3〜7個** — 破ってはならない原則。必ず**検証可能な禁止/保証**の形に言い換える。「高速であること」は絶対条件にならない（検証不能）。「バックグラウンドプロセスを持たない」「設定ファイル以外の永続状態を作らない」なら CI やレビューで機械的に検査できる
3. **KPI（優先順位付き）** — 数値目標とともに、確実に計測できるもの（hard: CI で fail させる）と環境依存でノイジーなもの（soft: 警告のみ + ローカル正式手順を文書化）に分類する
4. **MVP スコープ** — 「含む」と「含まない」の両方を列挙する。「含まない」リストが、将来のエージェントによるスコープ膨張を防ぐ
5. **技術スタック** — 言語・プラットフォーム・配布形態。絶対条件と KPI を満たせる根拠も添える

ユーザーと対話できる場合は一点ずつ確認する。自律実行中などで確認できない場合は依頼文から草案を導出し、PRD の該当箇所に `<!-- 要確認: ユーザー未承認 -->` を付けて完了報告で明示する。

## Phase 1: 文書層

### docs/PRD.md — single source of truth

`references/templates.md` の PRD 骨子に従って書く。守るべき性質:

- **全セクションに §番号を振る。** §番号はコード・テスト・CI・issue から参照される「番地」になる。番地がないと双方向トレーサビリティ（Phase 3）が成立しない
- 絶対条件・KPI・スコープ外リストを必ず独立セクションにする
- 仕様の解釈に迷ったら PRD が正、という地位をリポジトリ内で一意にする

### CLAUDE.md — エージェント向け圧縮

PRD をエージェントが毎回読むにはコストが高い。CLAUDE.md は PRD の**作業時チェックリストへの圧縮**であり、新情報を含めてはならない。守るべき性質:

- 絶対条件を冒頭近くに置き、「実装提案・レビュー時は常にこれに反していないか確認する」と明記する
- KPI の優先順位を固定順で列挙する
- 「コアに入れて良いもの / いけないもの」を具体的に列挙する（KPI を悪化させる要素の外出し原則）
- 変更前チェックリスト（KPI 悪化しないか、絶対条件に反しないか、外出しできないか、永続状態を持ち込まないか）
- コミット規約とリリースフロー（Phase 5 の内容）
- 末尾に「詳細仕様は docs/PRD.md を参照。衝突したら PRD が正」と書く

## Phase 2: リポジトリ骨格と隔離層

1. 言語標準の方法でプロジェクトを初期化する（git init、言語のプロジェクト初期化コマンド、.gitignore）
2. `assets/require-worktree.sh` を `.claude/hooks/` に、`assets/settings.json` を `.claude/settings.json` としてコピーする。この hook は primary worktree（main のチェックアウト）での編集ツール実行をブロックし、エージェントに作業用 worktree の作成を強制する。**なぜ**: 並列に動く複数エージェントが main を直接汚すと、失敗作業の巻き戻しと衝突解決が人間の仕事になる。worktree 隔離なら失敗したブランチごと捨てられる
3. hook メッセージ内のリポジトリ名プレースホルダを実名に置換する
4. `.claude/skills/` に PR ワークフロースキルの雛形を置く（`references/templates.md` の「オーケストレーションスキル」節）。内容: worktree 隔離 → 実装 → ローカル検証 → commit → ドラフトPR → 検品 → Ready 化、修正ループ上限付き

## Phase 3: テスト・トレーサビリティ層

実装ファイルごとに、隣に `<モジュール名>.test.md` を置く。これはテストコードではなく**テスト仕様表**で、3列の Markdown テーブル（検証事項 | 内容 | 備考）に各テストを1行で載せ、備考に対応する PRD §番号や issue 番号を書く。

**なぜ**: テストコードだけだと「このテストは何の仕様を守っているのか」が消える。仕様表があると、エージェントがテストを壊したとき・消したとき、どの仕様への違反かが diff レビューで即座に分かる。逆に PRD を変えたとき、影響するテストを表から逆引きできる。

立ち上げ時点では、絶対条件のうちテスト可能なものから最初のテストを書き、対応する `.test.md` 行を作る。空の雛形だけ置いて終わらせない（1行も実例がない表は誰も埋めない）。

## Phase 4: CI ゲート層

原則: **すべての KPI にゲートを対応させる。** 確実に測れるものは hard fail、ノイジーなものは soft warn + ローカルでの正式検証手順を文書化する。CI 構成に最低限含めるもの:

- lint **警告ゼロポリシー**（警告をエラー扱いにする）と format チェック。なぜ警告ゼロか: エージェントは「既存の警告 30 件」と「自分が増やした警告 1 件」を区別できないため、ゼロ以外の基準は維持不能
- 対象プラットフォーム全部でのビルド + テスト（matrix）
- 最低サポートツールチェーンバージョンの検証（依存更新が要求バージョンを静かに引き上げるのを検知する）
- KPI 計測 job — 例: 性能 KPI なら計測ツールで実測し、目標値とは別に**マージン付きの fail 閾値**を設ける（CI ランナーのノイズで flaky にしないため）。サイズ KPI なら成果物の実測値をレポートする
- 同一ブランチの古い実行を自動キャンセルする設定（エージェントは push 頻度が高い）

具体的な job 構成の考え方は `references/architecture-book.md` の「CI ゲート層」を参照。

## Phase 5: リリース層

特定のツール名を前提にせず、対象言語のエコシステムで**次の性質を満たす仕組み**を探して採用する:

1. コミットメッセージ規約（Conventional Commits）から次バージョンを自動算出する
2. バージョン bump + CHANGELOG + lockfile 同期を**リリース PR** として自動で開く（レビューゲート）
3. レジストリへの publish は**人間の明示トリガー**でのみ実行される（2段階目のゲート）

**なぜ2段階か**: パッケージレジストリへの公開は多くの場合不可逆。リリース PR のマージ（可逆）と publish（不可逆）を分離し、不可逆操作だけを人間ゲートにする。

**なぜ手動 bump 禁止か**: バージョン・タグ・lockfile をツールの所有物にしないと、手作業との競合でリリースが壊れる。エージェントには「適切な Conventional Commit を積むだけでバージョンは上がる」と CLAUDE.md で教える。

採用した仕組みの bump 規則（どの prefix がどの bump になるか）を表にして CLAUDE.md に書く。

## Phase 6: 検収（self-check）

生成完了後、必ず以下を**実際に確認**する（ファイルの存在確認ではなく動作確認）:

- [ ] 全絶対条件が PRD に §番号付きで存在し、CLAUDE.md に転記されている
- [ ] 全 KPI に CI ゲート（hard/soft）または文書化された手動検証手順が対応している
- [ ] primary worktree でファイル編集を試みると hook にブロックされる
- [ ] lint / format / テストがローカルで通る
- [ ] 最初のコミットがコミット規約に従っている
- [ ] PRD と CLAUDE.md に矛盾がない（CLAUDE.md に PRD にない新情報がない）

## 完了報告

ユーザーへの報告には次を含める: 作成したリポジトリのパス、絶対条件と KPI の一覧（要確認マークがあれば明示）、各 KPI とゲートの対応表、リリースフローの3ステップ、ユーザーが次にやること（リモート作成、CI シークレット設定など、エージェントには出来ないこと）。
