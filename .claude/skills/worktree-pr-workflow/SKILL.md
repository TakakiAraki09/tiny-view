# Worktree → PR Workflow

実装タスクを受け取り、**作業環境の隔離からPR提出可能な状態まで**を一気通貫で実行するワークフロースキル。

主担当（main process）はPM/Tech Lead として全体整合・品質・進捗に責任を持ち、独立した作業単位はsubagentへ委譲する。検品（要件充足チェック）に通って初めてPRをReady化し、通らなければ自動で修正ループを回す。

## このスキルが扱うもの・扱わないもの

- **扱う**: 「タスクを実装して、提出できる状態のPRにするまで」をまとめて任される依頼。worktree隔離・実装・コミット・PR作成・検品・修正ループを含む一連の流れ。
- **扱わない**:
  - 既にコミット済みの変更を単にPRにするだけ
  - コミットメッセージを整えるだけ
  - 調査・設計相談のみ（実装しない）
- 依頼の粒度が曖昧なときは短く確認する。例:「これは実装からPR提出まで一括で回す依頼でよいですか？」

## 絶対原則（破ってはならないガードレール）

これらはワークフローの安全性の核なので、毎フェーズで意識する。

- **隔離して作業する**: 必ず `git worktree` で別ディレクトリを切ってから実装する。現在のチェックアウト（特に `main`/`master`）上で直接編集しない。理由 — ユーザーの作業ツリーを汚さず、失敗しても捨てられる状態を保つため。
- **検品に通るまでドラフト**: PRは必ず `--draft` で作る。検品4軸すべてがOKになって初めて `gh pr ready` でReady化する。理由 — 未検証のものをレビュー可能状態にしてレビュアーの時間を奪わないため。
- **無限ループを作らない**: 修正ループには必ず上限（既定5回）と行き詰まり検知を設ける。理由 — 収束しない問題で延々とトークンを消費し続けないため。詳細は「修正ループ」節。
- **プロジェクト規約に従う**: 作業対象リポジトリに `CLAUDE.md` / `docs/PRD.md` があれば必ず読み、その絶対条件を検品基準に組み込む。理由 — 汎用的な「動けばOK」では各プロジェクト固有の禁止事項（例: TinyViewの No Server 原則）を見逃すため。
- **不可逆操作の前に状態を確認**: `git push` や `gh pr create` の前に、ブランチ・base・差分が意図通りかを確認する。base branch やブランチ名に迷ったらユーザーに一度だけ確認する。

## ワークフロー全体像

```
Phase 0  前提確認 ── タスク要件の明確化 / リポジトリ規約読込 / gh認証確認
Phase 1  Worktree作成 ── ブランチ命名 / git worktree add
Phase 2  実装 ── Explore調査 → expert実装委譲（main processが統合）
Phase 3  ローカル検証 ── テスト / ビルド / lint
Phase 4  Commit ── /commit の規約に準拠
Phase 5  ドラフトPR作成 ── /pr の流れに準拠（必ず --draft）
Phase 6  検品 ── 4軸チェック（独立レビューエージェント含む）
Phase 7  判定 ── OK → gh pr ready / NG → Phase 2へ戻り修正ループ
Phase 8  完了報告 ── PR URL提示 / worktreeの後始末方針
```

修正ループは Phase 6→7→（NGなら）2 を回す。Phase 1 の worktree とブランチはループ中ずっと使い回す（毎回切り直さない）。

## Phase 0: 前提確認

実装に入る前に、後戻りを減らすための前提を固める。

1. **タスク要件を1〜3行で言語化する**。受け取った依頼を「何を満たせば完了か」の受け入れ条件として書き出す。これが後の検品（軸②）の判定材料になるので、曖昧なら今のうちにユーザーへ一点確認する。
2. **リポジトリ規約を読む**。`CLAUDE.md` / `docs/PRD.md` / `README` の有無を確認し、あれば絶対条件・禁止事項を抽出して検品チェックリストに加える。
3. **検証コマンドを把握する**。`package.json` の scripts / `Cargo.toml` / `Makefile` / CI設定（`.github/workflows`）を見て、テスト・ビルド・lint・型チェックの実行コマンドを特定する。なければ「検証手段なし」として軸①は手動確認に切り替える旨を記録する。
4. **base branch を決める**。既定は `main`（リポジトリのデフォルトブランチ）。派生元が別ブランチなら確認する。
5. **gh の認証を確認する**（`gh auth status`）。未認証ならユーザーに `! gh auth login` を促す。

## Phase 1: Worktree作成

- ブランチ名はタスク内容から conventional に生成する: `feat/<要約>` / `fix/<要約>` / `chore/<要約>` 等。要約はケバブケースの短い英語。
- worktree は対象リポジトリの外（または `../`）に切るのが安全。例:

```bash
git worktree add ../<repo>-<branch-summary> -b feat/<branch-summary> <base>
```

- 以降の全作業はこの worktree ディレクトリ内で行う。パスを記録し、Phase 8 まで使い回す。

## Phase 2: 実装

プロジェクトの CLAUDE.md 方針に従い、main process は抱え込まず委譲する。

1. **調査**: Explore エージェントで関連コード・既存パターン・影響範囲を把握する。
2. **実装**: 独立した実装単位を該当エキスパート（typescript-expert / rust 等、言語に応じて）へ委譲する。複数の独立単位は並列に委譲してよい。
3. **統合**: main process が成果物・前提・判断理由・懸念点を受け取り、整合性をレビューして統合する。subagent の出力をそのまま採用しない。

実装は worktree 内で完結させる。

## Phase 3: ローカル検証

commit前に、Phase 0 で特定した検証コマンドを実行する。

- テスト / ビルド / lint / 型チェックを実行し、結果を記録する。
- 失敗が出たら**この場で直す**（Phase 6 まで持ち越さない）。ここを通さずにコミットしない。
- 検証手段が無いプロジェクトなら、その旨を明記して次へ進む。

## Phase 4: Commit

- 既に worktree 上の feature ブランチにいるので、ブランチ作成プロンプトは不要。
- 複数の論理変更があれば分割コミットする。

## Phase 5: ドラフトPR作成

1. `git push -u origin <branch>` でリモートへ push。
2. `gh pr create --draft --base <base> --title "<English title>" --body "<body>"` でドラフトPRを作成する。
   - タイトルは英語、本文は `/pr` の日本語フォーマット（## 概要 / ## 詳細 / ## 補足）。
   - PR本文に Claude の署名を入れない。
3. 作成したPRの番号・URLを記録する。

## Phase 6: 検品（4軸チェック）

PRの差分が**要件を満たしているか**を4軸で厳格に判定する。各軸の結果を「PASS / FAIL（理由・該当箇所）」で記録する。

| 軸                     | 内容                                     | 判定方法                                                                                                  |
| ---------------------- | ---------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| ① テスト/ビルド/Lint   | Phase 0 で特定した検証コマンドが全て通る | コマンドを実際に実行して結果で判定                                                                        |
| ② 元タスクの要件充足   | Phase 0 で言語化した受け入れ条件を満たす | 差分とタスク要件を突き合わせてレビュー                                                                    |
| ③ 独立レビュー         | 第三者視点でのコード品質・バグ           | `code-review` スキル or 独立した code-quality-reviewer エージェントを走らせ、**重大な指摘がゼロ**かで判定 |
| ④ プロジェクト規約整合 | CLAUDE.md / PRD の絶対条件に反していない | Phase 0 で抽出した禁止事項チェックリストと照合                                                            |

- 軸③のレビューは**必ず別エージェントに委譲**する（実装した本人＝main processの自己レビューは盲点が残るため）。指摘は重大度を分類し、「重大（バグ・規約違反・要件未達）」が残る間は FAIL とする。軽微な好みの指摘は記録するが FAIL にしない。
- **検品はワークフローの終端ではなく中間工程**。レビューエージェントの生出力（指摘リスト等）をそのまま最終成果物にして実行を終えてはいけない。委譲した結果は main process が受け取り、4軸の判定表にまとめ直してから必ず Phase 7→8 へ進む。理由 — 検品結果で会話を打ち切ると、Ready化判定・PR成果物の提示・後始末の報告（Phase 7/8）が丸ごと抜け落ち、ユーザーは「何が起きたか」を受け取れないため。
- 全4軸 PASS なら Phase 7 で OK 判定。1つでも FAIL なら NG 判定。

## Phase 7: 判定

**全軸 PASS（OK）の場合:**

```bash
gh pr ready <pr-number>
```

でドラフトを解除し Ready for review にする。これでワークフロー完了 → Phase 8。

**FAIL が残る（NG）の場合:**

修正ループに入る。FAIL の指摘を修正タスクに変換し、Phase 2（実装）へ戻る。worktree とブランチはそのまま使い回す。修正後 Phase 3→4（追加コミット）→ 既存PRはpushで自動更新 → Phase 6（再検品）。

## 修正ループ（停止条件）

NG のたびに無条件でループせず、必ず収束管理する。

- **イテレーション上限**: 既定 **5回**。各ループの冒頭で `iteration N/5` を宣言する。
- **行き詰まり検知**: 各イテレーションの FAIL 指摘を記録し比較する。**同じ指摘が2回連続で残った（＝直前ループで直そうとしたが解消しなかった）**ら、それ以上同じ手は繰り返さない。
- **どちらかに達したら中断してエスカレーション**: ループを止め、ユーザーに以下を整理して報告する。
  - 現状どこまで満たし、どの軸の何が残っているか
  - これまで試した修正アプローチと、なぜ収束しなかったかの仮説
  - 次の判断の選択肢（例: アプローチを変える / 要件を見直す / 一旦ドラフトのまま保留）
- 中断時もPRは**ドラフトのまま残す**（Ready化しない）。捨てずに残すことで、ユーザーが続きを引き取れる。

理由 — 収束しない問題に延々とトークンを溶かすより、行き詰まりを早く可視化してユーザーの判断を仰ぐ方が価値が高い。同じ修正を繰り返すのは「アプローチ自体が間違っている」サインなので、回数ではなく*手詰まり*を検知して止めるのが要点。

## Phase 8: 完了報告（必須の締め）

検品が PASS でも FAIL（中断）でも、**ワークフローは必ずこのフェーズで締める**。検品やレビューの出力で会話を終えず、最後に必ず以下を構造化して提示する。

- 結果: Ready化できたか / 中断したか
- PR URL と番号
- 検品4軸の最終結果サマリ
- worktree の後始末方針: `git worktree remove <path>`（マージ後でよい旨を添える。ユーザーが続きを触る可能性があるので**勝手に削除しない**）

## ドライランモード（評価・テスト用）

このスキルは副作用（push / PR作成 / Ready化）を伴うため、評価ハーネスやテスト時は**ドライラン**で回す。依頼やテストプロンプトに「ドライラン」「dry-run」「実PRは作らない」等の指示があれば、以下に切り替える。

- **対象**: リモートへ作用する操作のみを「ログ出力に置換」する。具体的には `git push` / `gh pr create` / `gh pr ready` / `gh pr edit` を実行せず、**実行予定のコマンドを `DRYRUN:` プレフィックス付きで出力**する。
- **そのまま実行する**: worktree作成・編集・ローカル検証（テスト/ビルド/lint）・commit は実環境で行ってよい（ローカル完結のため）。テスト用に使い捨てのローカルgitリポジトリが与えられた場合はその中で完結させる。
- **検品**: 4軸とも通常どおり実行する。push しない代わりに、PR本文・タイトル・差分をログに出して「PRに相当する成果物」を提示する。
- **判定とループ**: 通常どおり行うが、Ready化は `DRYRUN: gh pr ready ...` のログ出力に置換する。

ドライランの目的は「ワークフローの判断ロジック（隔離・検証・検品・ループ収束）が正しく機能するか」を、GitHub を汚さずに検証することにある。
